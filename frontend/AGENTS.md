# DG Technical Test - Frontend AI Agents Manual

## 1. Technologies & Libraries
- **Framework:** React 19 + TypeScript
- **Bundler:** Vite
- **Routing:** `@tanstack/react-router` (Type-safe, file-based routing)
- **Styling:** Tailwind CSS v4
- **UI Components:** **shadcn/ui** (Use this for all base components like buttons, forms, inputs, dialogs, etc.)
- **Authentication:** **Better Auth Client** (To interface with the backend Better Auth instance seamlessly)
- **Icons:** `lucide-react`
- **Testing:** Vitest + React Testing Library + jsdom

## 2. Frontend Commands
Execute these inside the `frontend/` directory or using `bun -F frontend <command>` from root:
- **Dev Server:** `bun run dev` (Runs on port 3000)
- **Build:** `bun run build`
- **Run all tests:** `bun run test` (Vitest)
- **Run a single test:** `bunx vitest run src/path/to/component.test.tsx`
- **Watch a single test:** `bunx vitest src/path/to/component.test.tsx`

## 3. Design Patterns & Architecture

**UI & Componentization:**
- Heavily rely on **shadcn/ui** for the visual layer. Keep UI elements small, reusable, and accessible.
- Use functional React components with Hooks. Extract complex logic into custom hooks (e.g., `useUserForm`, `useAuth`).

**Routing & Data Fetching:**
- Leverage TanStack Router's loaders and actions for data fetching and mutations, keeping component trees clean.
- Use the `.gen.ts` files automatically generated by the router plugin for type-safe links and navigation.

**Authentication (Better Auth Client):**
- Use the Better Auth client hooks and methods (`useSession`, `signIn`, `signOut`) to manage sessions, login state, and UI access control on the client side.
- Read tokens or cookies securely as provided by Better Auth.

**Forms & State Management:**
- Forms must feature strict validation (e.g., **Zod**) providing immediate feedback on invalid fields (CPF, CNPJ, CEP).
- Combine Zod schemas with shadcn/ui Form components (`react-hook-form` + `@hookform/resolvers/zod` is highly recommended).

**Error Handling:**
- Always handle API failures gracefully. Display clear, translated error messages to the user detailing exactly what failed and why using toasts or inline alerts.

## 4. Code Style & Naming
- Components: `PascalCase.tsx` or `kebab-case.tsx`, stick to the router's convention for page files.
- Utilities: `camelCase.ts` or `kebab-case.ts`.
- Always use absolute paths (e.g., `#/*` via `imports` in `package.json`) where configured.
